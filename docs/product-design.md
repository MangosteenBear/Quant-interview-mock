> **项目名称**：量化面试刷题题库平台
> **文档版本**：v1.0（一期 MVP 设计）
> **更新日期**：2026-06-23

## 一、产品定位

### 1.1 一句话定位
面向量化岗求职者的**轻量化刷题工具**，移动端优先，Web + 微信小程序双形态，一套数据两套前端。

### 1.2 目标用户
| 用户群 | 核心诉求 | 使用场景 |
|--------|---------|---------|
| 在校金融/数学/计算机应届生 | 量化岗面试突击，覆盖数理+编程题型 | 面试前专项训练、碎片化刷题 |
| 转行量化求职者 | 系统补齐量化知识体系，查漏补缺 | 电脑集中模考、手机通勤刷题 |
| 在职量化从业者 | 复盘面试、保持手感 | 碎片化复习 |

### 1.3 核心使用场景
1. **手机碎片化刷题**：通勤/排队时单题速刷，题干置顶、答案折叠，减少跳转
2. **电脑集中模考**（二期）：大屏套卷限时作答，模拟真实面试节奏
3. **面试前专项突击**：按知识点标签/难度筛选，定向突破弱项

### 1.4 与竞品的差异化
- **轻量化**：无需下载重型 App，Web + 小程序即开即用，低加载、无广告
- **题型全覆盖**：数理统计、概率、金融衍生品、C++/Python 量化编程、算法、brain teaser、机器学习
- **公式友好**：原生 LaTeX 渲染，量化公式不丢真
- **数据源专业**：以经典量化面试题库电子书为基础数据源（绿皮书、红皮书、Heard on the Street 等 8 本）

## 二、技术栈选型

> 选型原则：**轻量化低成本，优先成熟简易技术栈，避免重型复杂框架，优先保证移动端加载速度与兼容性。**

### 2.1 技术栈总览

| 层 | 技术选型 | 备选 | 选型理由 |
|---|---|---|---|
| 前端框架 | **uni-app (Vue3 + Vite + TS)** | Taro | Vue 生态 + 双端适配成本最低，单人维护友好，条件编译 `#ifdef` 简洁 |
| 状态管理 | **Pinia** | Vuex | Vue3 官方推荐，TS 友好，API 更简洁 |
| 公式渲染 | **KaTeX** | MathJax | bundle 仅 28KB(gzip)，同步渲染快，移动端性能优；MathJax 1MB+ 易卡顿 |
| 后端 | **Python 3.11 + FastAPI** | — | OCR/数据流水线天然 Python，前后端语言统一，OpenAPI 文档自动生成 |
| ORM/迁移 | **SQLAlchemy 2.0 (async) + Alembic** | Tortoise | 成熟、异步、类型友好 |
| 数据库 | **PostgreSQL 16** | — | FTS + JSONB + 关系建模一站式，万~十万级题库足够 |
| 缓存 | **Redis 7** | — | 计时/会话/热门缓存/限流 |
| 全文搜索 | **PostgreSQL FTS + pg_jieba** | Elasticsearch | PG 内置全文搜索够用，避免 ES 单人运维负担 |
| OCR 文字 | **PaddleOCR PP-OCRv4** | Tesseract | 中英文混合影印模糊文字效果好 |
| OCR 公式 | **PaddleOCR PP-FormulaNet-S** (+ UniMERNet) | MathPix(付费) | 免费、5万 LaTeX 词汇、500万数据训练，零成本替代 MathPix |
| 版面分析 | **PP-StructureV3** | LayoutLMv3 | 文字/公式/表格/多栏区域自动切分 |
| PDF 解析 | **PyMuPDF(fitz) + pdfplumber** | — | 渲染+文本+表格+多栏组合 |
| 去重 | **SimHash + MinHash(LSH)** | — | 题干相似度比对，汉明距离判重 |
| 部署 | **单 VPS + Docker Compose** | Serverless | 低成本、可控、OCR 离线本地跑 |
| 反代/静态 | **Nginx** | Caddy | 成熟稳定 |

### 2.2 关键决策论证

**① uni-app vs Taro → 选 uni-app**
uni-app 基于 Vue3，单人维护下 Vue 模板更直观、中文资源丰富（DCloud 社区）、HBuilderX 对小程序编译成熟。双端条件编译 `#ifdef MP-WEIXIN / H5` 一套代码两套产物。Taro 优势在 React Native 多端，但本项目只需 H5 + 微信小程序，用不到 RN。

**② OCR 方案 → PaddleOCR 全家桶，不依赖 MathPix**
- **PP-FormulaNet-S**：百度飞桨自研，500 万数据训练，5 万 LaTeX 词汇，免费开源，印刷体公式识别效果好
- **UniMERNet**：复杂公式（长公式、矩阵、嵌套分式）的补充识别
- **影印模糊 PDF 策略**：PyMuPDF 300DPI 渲染 + OpenCV 图像增强（去噪/二值化/锐化）→ 文本层稀疏判为扫描版全走 OCR → PP-OCRv4 对模糊中英文有专门优化
- MathPix 付费且限流，仅作为复杂公式的"人工兜底参考"

**③ 公式渲染 → KaTeX + 双端差异化**
- H5 端：KaTeX 直接渲染（28KB，同步渲染，移动端流畅）
- 小程序端：KaTeX 无法直接运行，方案是**后端预渲染 HTML**（LaTeX → HTML 片段下发，小程序 rich-text / mp-html 渲染）。这是双端差异化的必要妥协

**④ 数据库 → PostgreSQL，FTS 不上 Elasticsearch**
题库规模预估 8 本书 × 数百~数千题 = 万级，长期十万级。PG FTS + pg_jieba 中文分词 + GIN 索引完全覆盖。Elasticsearch 单人运维成本高（JVM、集群、内存），过度设计。

## 三、数据处理流水线设计

> 核心难题：影印版 PDF → 结构化题库。PDF 存在文字模糊、排版错乱、公式图片、多栏排版、答案分离等问题。

### 3.1 流水线总览

```
PDF ─► 页面渲染 ─► 版面分析 ─► 区域切分 ─► 分流OCR ─► LaTeX重建
     │                                          (文字/公式/表格)
     └► 文本层提取(判断扫描版)                          │
                                                      ▼
                题目边界识别 ─► 答案关联 ─► 去重 ─► 校验 ─► 人工审核 ─► 入库
```

### 3.2 各环节具体方案

#### Step 1 — PDF 预处理与扫描版判定
- PyMuPDF 渲染每页为 300DPI PNG；同时 `page.get_text()` 提取原生文本层
- 判定逻辑：文本层字符数 / 页面面积 < 阈值 → 影印扫描版 → 全页走 OCR；否则混合（文本层 + OCR 补充）

#### Step 2 — 版面分析（PP-StructureV3）
- 识别区域类型：`text / title / formula / table / figure / header / footer`
- **多栏处理**：区域坐标按 x 聚类分栏，栏内按 y 排序重组阅读顺序（解决双栏排版）
- 剔除页眉页脚（页码、书名重复出现）

#### Step 3 — 分流 OCR
| 区域类型 | OCR 引擎 | 输出 |
|---------|---------|------|
| text | PP-OCRv4（中英文混合） | 纯文本 |
| formula | PP-FormulaNet-S → 低置信度走 UniMERNet | LaTeX |
| table | PP-Structure 表格识别 | HTML/Markdown |

#### Step 4 — LaTeX 重建
- 文本与公式 LaTeX 拼接为 Markdown，公式用 `$...$`（行内）/ `$$...$$`（行间）包裹
- 统一中间格式：`{type, content_markdown, bbox, page}`

#### Step 5 — 题目边界识别
- 正则匹配题号模式：`^\s*(Q|Question|Problem|Ex\.|Exercise)?\s*\d+[\.\)、]`
- 结合章节标题切分题目块
- 多栏重组后按阅读顺序遍历，题号递增校验，断题（跳变/缺失）进异常队列

#### Step 6 — 答案关联（处理答案分离）
- 量化书常见"答案在书末/章末"。先全书跑流水线得题目池 + 答案池（答案区特征：`Answer/Solution` 开头 + 题号）
- 按题号 + 章节建立映射表关联题干与答案/解析
- 无法自动匹配的（题号错位、缺答案）→ 标记 `unmatched` 进人工队列

#### Step 7 — 去重机制
- 题干 LaTeX 归一化：去空格、统一符号（`\cdot`→`*`）、去题号、小写化
- **SimHash**（64位指纹）+ 汉明距离 ≤ 3 → 疑似重复候选
- **MinHash + LSH**（Jaccard 相似度 ≥ 0.85）二次确认
- 疑似重复进人工确认队列：合并为一题，多版本解析分别保留并标注来源

#### Step 8 — 数据校验
- LaTeX 语法校验：`latex2mathml` 或 KaTeX 解析校验，语法错误标记修复
- 必填字段校验：题干非空、题型合法、有答案或明确标记"无答案"
- 题型分类：选择题（有选项）/ 填空题 / 简答题 / 证明题（规则 + 人工）

#### Step 9 — 人工审核后台
- FastAPI + 简易 Vue 页面：逐题展示 OCR 结果 + 原始 PDF 截图对照
- 人工编辑题干/答案/标签/难度/岗位
- 状态机：`pending → reviewing → revised → published / rejected`
- 审核通过才进正式题库表

### 3.3 入库数据结构（核心表）

```sql
-- 题目主表
questions (
  id, stem_markdown, stem_latex_normalized,   -- 题干(Markdown含LaTeX) + 归一化指纹
  question_type,                                -- choice/fill/short/proof
  difficulty,                                   -- 1-5
  simhash,                                      -- 64位去重指纹
  source_id, book_page, book_chapter,           -- 来源追溯
  status,                                       -- pending/reviewing/published
  ingested_at, updated_at
)

-- 选项表（选择题）
options (id, question_id, label, content_markdown, is_correct)

-- 解析表（1题N版本解析，支持相同题干合并保留多来源解析）
solutions (id, question_id, content_markdown, source_id, version)

-- 标签表 + 关联（多对多：知识点/岗位/题型）
tags (id, name, type)                            -- type: knowledge/position/topic
question_tags (question_id, tag_id)

-- 来源表（数据溯源）
sources (id, book_title, author, edition, file_hash, notes)

-- 作答记录（一期匿名，设备维度）
attempt_logs (id, device_id, question_id, answer, is_correct, duration_ms, created_at)

-- 收藏
favorites (id, device_id, question_id, created_at)
```

### 3.4 数据处理强制规范（合规约束）
1. **标准化结构化**：每题拆分题干、选项、标准答案、详细解析、题型标签、难度、对应岗位、来源标注
2. **公式统一 LaTeX**，PDF 模糊文字做 OCR 纠错
3. **剔除重复/残缺/无答案无效题目**，相同题干自动合并保留多版本解析
4. **每条数据标记数据源、入库时间**
5. **版权约束**：仅处理公开影印/免费分享资料，不破解付费文档，不商用他人原创付费题库，禁止二次分发售卖题库数据

## 五、核心页面与交互逻辑

### 5.1 页面结构（双端通用）

```
首页（题库入口）
├── 题库列表页（按书/章节浏览）
│   └── 题目详情与作答页
│       ├── 题干区（置顶，LaTeX 渲染，长题滚动）
│       ├── 选项区（选择题）/ 作答区（填空/简答）
│       ├── 答案判定（即时反馈）
│       └── 解析区（折叠展开）
├── 搜索页
├── 收藏列表页
└── 设置页（夜间模式/字体大小）
```

### 5.2 刷题页交互逻辑（核心）

| 交互要素 | 设计规则 |
|---------|---------|
| 题干区 | 置顶，LaTeX 公式自适应缩放，长题目支持滚动查看 |
| 答案/解析 | 默认折叠，点击展开，减少页面跳转 |
| 作答判定 | 选择/填空即时判对错；简答/证明展示参考答案 |
| 答题计时 | 进入题目自动计时，提交作答时上报 duration_ms |
| 收藏 | 右上角星标，点击切换，设备维度持久化 |
| 夜间模式 | 全局 CSS 变量切换，刷新保持设置 |
| 字体调节 | 3 档（小/中/大），影响题干与解析，持久化 |

### 5.3 视觉风格
- **低饱和商务简约风**，主色调沉稳专业（建议深蓝/墨绿系）
- 难度标签用浅色系区分（P1 浅绿 / P2 浅黄 / P3 浅橙 / P4 浅红 / P5 深红）
- 不使用花哨动效，优先保证阅读舒适
- 移动端竖屏适配，无冗余广告、复杂弹窗

## 六、指标体系（一期轻量版）

> 单人项目不堆重型埋点，聚焦"题库质量"与"刷题核心体验"两类指标，后端日志 + 简单统计即可采集。

| 指标层级 | 指标 | 定义 | 一期目标/基线 | 采集方式 |
|---|---|---|---|---|
| **北极星** | 周有效刷题量 | 每周用户实际作答（非空提交）的题目总数 | 上线首月 ≥500 | attempt_logs 聚合 |
| **题库质量（健康）** | 题库可用题量 | status=published 的题目数 | 一期 ≥1000 | questions 表 count |
| | 审核通过率 | published / (published+rejected) | ≥80% | 审核后台日志 |
| | OCR 公式识别准确率 | 人工抽检公式正确率 | ≥85% | 抽样人工核对 |
| | 重复题合并率 | 被去重合并的题 / 总题 | 监控异常波动 | dedup 日志 |

| | 平均单题作答时长 | attempt_logs.duration_ms 均值 | 监控异常（过短=乱答/过长=题目难） | 后端统计 |
| | 收藏率 | 被收藏题数 / 总作答题数 | 监控偏好 | favorites 表 |
| | 搜索命中率 | 搜索有结果的请求 / 总搜索 | ≥70% | 搜索 API 日志 |

> 二期补充账号体系后，再上"用户留存（次日/7日）""错题复习完成率""模考完成率"等用户级指标。

## 八、风险与合规

### 8.1 版权合规风险（高）
8 本 PDF 需确认来源授权。项目指令明确"仅处理公开影印/免费分享资料，不破解付费文档"。其中绿皮书、红皮书等为正式出版物，**仅限个人学习刷题使用，禁止二次分发售卖题库数据**。

**规避方案**：上线前完成版权合规自查；公开题库内容建议只展示题干 + 自主编写解析，不直接复制原书答案原文。

### 8.2 OCR 识别误差风险（中）
影印模糊 PDF + 数学公式识别存在误差，公式识别准确率约 80-90%。

**规避方案**：人工审核后台作为必要环节；复杂公式人工兜底；逐步积累标注数据微调模型。

### 8.3 小程序审核合规风险（中）
教育类小程序需相应资质，内容涉及时政/金融需确认类目。

**规避方案**：提审前确认类目与内容合规，题库内容定位为"职业技能学习"。

### 8.4 OCR 算力风险（低）
CPU 跑 PaddleOCR 较慢（单本 PDF 可能数小时）。

**规避方案**：离线批处理不阻塞生产；如有 GPU 本机可加速；分批跑批。

## 九、项目目录结构（实施阶段）

